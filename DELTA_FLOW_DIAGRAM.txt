                         FINAL MESSAGE (No Throttle)
════════════════════════════════════════════════════════════════════════════════

When LLM finishes, lifecycle event with phase: "end" triggers:

    emitChatFinal(sessionKey, clientRunId, sourceRunId, seq, "done")
           ↓
    Retrieve bufferedText from chatRunState.buffers (all accumulated text)
           ↓
    Send FINAL state message (bypasses 50ms throttle)
           ↓
    Clean up: buffers.delete(), deltaSentAt.delete()

This ensures the COMPLETE response reaches the client.


════════════════════════════════════════════════════════════════════════════════
                         KEY STATE STRUCTURES
════════════════════════════════════════════════════════════════════════════════

ChatRunState (in server-chat.ts):
  {
    deltaSentAt: Map<clientRunId, timestampMs>,    ← Tracks last delta time
    buffers: Map<clientRunId, accumulatedText>,    ← Holds throttled text
    registry: ChatRunRegistry,                      ← Tracks active runs
    abortedRuns: Map<runId, timestampMs>           ← Tracks cancellations
  }

Per-Client Throttling:
  • Each clientRunId gets its own timestamp in deltaSentAt
  • Multiple concurrent users don't interfere with each other
  • Each has independent 50ms throttle window
